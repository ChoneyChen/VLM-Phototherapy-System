# 架构规范文档 - VLM Phototherapy System

## 一、系统目标

本项目用于构建一个生产级的多用户面部皮肤识别与智能光疗系统，核心目标如下：

- 使用云端视觉模型完成面部皮肤状态识别
- 为每位用户沉淀独立的每日图像与面部状态档案
- 将“面部识别”和“治疗方案生成”彻底解耦
- 以整体智能光疗面罩为控制目标，而不是零散的单区域设备
- 为未来真实硬件接入预留稳定、科学、可扩展的控制协议

当前系统已经明确分成四个业务阶段：

1. 面部测评 `assessment`
2. 面部档案 `archive`
3. 治疗方案 `treatment plan`
4. 治疗控制与治疗记录 `treatment control / treatment records`

其中：

- 面部测评负责“识别和提取结构化信息”
- 治疗方案负责“基于档案、资料库和知识库生成可执行策略”
- 治疗控制负责“把策略映射为智能面罩控制会话”
- 治疗记录负责“保存某次治疗启动时的控制快照与状态”

## 二、统筹修改原则

所有功能修改必须先从前端、后端、共享合同三个维度统筹规划，再开始实现。

修改前必须先明确：

- 数据从哪里产生
- 数据由谁持久化
- 哪一层负责归一化
- 前端状态如何切换
- 是否需要新增共享合同、知识库或提示词模板
- 是否需要同步修改 README、API 文档与本架构文档

禁止以下情况：

- 只改前端页面，不同步后端 contract
- 只改后端接口，不同步前端状态流
- 直接把模型自由文本作为硬件控制字段
- 把硬件协议细节直接散落在页面组件中

## 三、总体分层

### 1. `backend/`

后端服务与 CLI。后端必须可以脱离前端独立运行。

职责：

- HTTP API
- CLI
- 业务规则
- 模型调用
- 数据库存储
- 图像归档
- 治疗方案生成
- 智能面罩控制会话生成
- 治疗记录落库
- 硬件协议抽象

### 2. `frontend/`

前端应用，只负责：

- 页面与交互
- 路由与工作流切换
- API 调用
- 多语言显示
- 本地 UI 状态

前端不得负责：

- 模型调用
- 数据库存储
- 硬件协议生成
- 治疗方案核心归一化

### 3. `common/`

前后端共享、与运行时无关的资产。

当前放置：

- 结构化合同 `contracts/`
- 提示词模板 `prompts/`
- 皮肤与治疗知识库 `knowledge/`
- 诊疗资料库占位 `knowledge/treatment_plan_library/`

### 4. `docs/`

对外文档层。

- `README.md` 项目总览
- `docs/API.md` 独立接口文档
- 本文件 `ARCHITECTURE.md` 作为强制架构规范

- 用户业务规则

- 治疗方案生成与规范化

### `backend/app/infrastructure/`

基础设施层。

包括：

- 数据库仓储

用于：

- API request / response schema
- CLI 输出结构
- 治疗控制和治疗记录结构

### `frontend/src/app/`

应用级壳层。

- 路由
- 当前用户绑定
- 页面间状态衔接
- 治疗方案选中态

### `frontend/src/pages/`

当前页面：

- 面部测评

- 治疗方案
- 治疗控制
- 治疗记录
- 用户管理

### `frontend/src/features/`

按业务模块拆分的前端功能。

例如：

- 测评表单
- 摄像头拍照弹窗
- 档案时间线
- 用户入口 Gate

### `frontend/src/shared/`

前端共享层。

- API client
- TypeScript 类型
- 全局样式
- i18n
- 通用布局

## 五、核心业务解耦

### 1. 面部识别与治疗方案必须解耦

面部测评阶段只负责：

- 识别整体状态
- 识别分区状态
- 提取问题类别
- 提取严重程度
- 返回测评阶段说明
- 生成用于归档的结构化结果

它不负责生成真正的智能面罩治疗方案。

治疗方案必须在档案页由“新增治疗方案”动作单独触发。

原因：

- 后续会引入更丰富的诊疗资料库和案例库
- 治疗方案可能需要多次生成、迭代或版本比较
- 识别结果应保持稳定，不应因为治疗策略变化而被重写
- 方案生成模型和识别模型可以独立演进

### 2. 治疗方案与治疗控制必须解耦

治疗方案回答的是：

- 这份档案对应的整体严重性是什么
- 哪些分区问题需要重点处理
- 智能面罩每个区域建议使用什么 LED 颜色
- 全局加热、加湿、亮度、时长的推荐起点是什么

治疗控制回答的是：

- 当前这次准备启动的面罩控制参数是什么
- 用户是否修改了推荐参数
- 最终下发给硬件的控制快照是什么

因此：

- 治疗方案是策略层
- 治疗控制是执行前配置层

### 3. 治疗控制与治疗记录必须解耦

治疗记录不是简单复用治疗方案。

治疗记录必须保存：

- 哪份治疗方案被使用
- 启动时的全局参数
- 启动时的分区 LED 颜色
- 启动时间
- 当前状态

这样后续才能支持：

- 开始 / 暂停 / 恢复 / 完成
- 真实设备执行回执
- 历史治疗追溯

## 六、智能光疗面罩控制模型

当前硬件抽象已经从“单区域设备”升级为“整体智能光疗面罩”。

### 1. 分区 LED 颜色

面罩各分区独立控制 LED 颜色。

标准分区如下：

- `Forehead Zone`
- `Periorbital Zone`
- `Nasal Zone`
- `Left Malar Zone`
- `Right Malar Zone`
- `Perioral Zone`
- `Mandibular/Chin Zone`
- `Jawline Zone`

当前每个分区仅开放：

- `led_color_code`

可选色值：

- `red`
- `orange`
- `yellow`
- `green`
- `cyan`
- `blue`
- `purple`

### 2. 全局参数

以下参数是整个智能面罩共享的全局参数，不再按区域拆开：

- `brightness_percent`
- `temperature_celsius`
- `humidification_frequency_level`
- `timer_minutes`

当前安全范围：

- 亮度：`0-100`
- 温度：`20-40`
- 加湿频率：`0-100`
- 定时：`1-60`

### 3. 控制操作

治疗控制页当前至少要支持：

- 开始治疗
- 暂停治疗
- 恢复治疗

后续将扩展：

- 完成治疗
- 停止治疗
- 设备连接状态
- 故障与告警回执

## 七、数据流

### 1. 面部测评数据流

1. 用户被选中或创建
2. 前端上传图片或摄像头拍照
3. 后端保存原始图像
4. 后端读取皮肤评估提示词与知识分类
5. 后端调用视觉模型
6. 后端归一化为结构化测评结果
7. 测评结果写入数据库
8. 前端展示结果并归档

### 2. 治疗方案生成数据流

1. 用户在面部档案页选择某份历史档案
2. 用户点击“新增治疗方案”
3. 前端显示假进度条
4. 后端读取该档案的结构化测评结果
5. 后端读取诊疗资料库、案例库、设备画像和输出合同
6. 后端调用 `qwen-plus`
7. 后端归一化生成结构化治疗方案
8. 治疗方案写入数据库
9. 前端进入治疗方案页展示方案卡

### 3. 治疗控制数据流

1. 用户在治疗方案页点击某张方案卡
2. 前端请求 `treatment control preset`
3. 后端根据治疗方案生成智能面罩控制会话
4. 前端载入人脸简图、分区颜色与全局参数
5. 用户修改参数
6. 前端创建治疗记录
7. 后端保存治疗记录与控制快照

### 4. 治疗记录数据流

1. 治疗开始时创建记录
2. 暂停 / 恢复时更新状态
3. 治疗记录页读取所有历史治疗记录
4. 后续真实设备回执将更新同一条记录或新增执行日志

## 八、命名约束

### 用户标识

- 数据库内部主键可用自增 ID
- 对外必须使用业务 ID
- 格式统一为 `USR-0001`

### 测评 ID

- 格式统一为 `ASM-xxxxxxxxxxxx`

### 治疗方案 ID

- 格式统一为 `TPL-xxxxxxxxxxxx`

### 治疗记录 ID

- 格式统一为 `REC-xxxxxxxxxxxx`

### 严重性

- `low`
- `medium`
- `high`

### 语言

- `en`
- `zh`

### 模型供应商

面部识别模型：

- `gemini`
- `qwen`

治疗方案生成模型：

- `qwen_plus`

## 九、多语言规则

多语言必须遵守以下原则：

- 内部结构化字段统一使用英文标准码
- 中文模式只影响页面文案与自由文本展示
- 数值字段不做本地化结构变更
- 硬件字段名必须保持英文稳定
- 中英文自由文本必须用 `LocalizedText`

特别说明：

- 中文模式时，模型返回的可读描述必须显示中文
- 英文模式时，模型返回的可读描述必须显示英文
- 对于颜色、严重性、分区名称等结构化字段，必须在前端维护稳定的中英文映射

## 十、知识库与资料库架构

为了支持后续更专业的方案生成，当前已经预留了独立的诊疗资料库目录：

- `common/knowledge/treatment_plan_library/case_memory.json`
- `common/knowledge/treatment_plan_library/mask_device_profile.json`
- `common/knowledge/treatment_plan_library/clinical_guidelines.md`
- `common/knowledge/treatment_plan_library/authoring_notes.md`

用途如下：

- `case_memory.json`
  - 用于沉淀可复用治疗案例
- `mask_device_profile.json`
  - 描述当前智能面罩允许的参数边界
- `clinical_guidelines.md`
  - 放置诊疗规则、禁忌与经验原则
- `authoring_notes.md`
  - 说明如何继续维护资料库

这些文件属于共享资产层，后续由你持续追加内容，后端提示词构建器会自动读取它们。

## 十一、硬件协议原则

### 1. 允许存在评估阶段建议，但不得直接驱动设备

面部测评结果里可以保留评估阶段的建议性字段，例如早期的 `phototherapy_command.v1`。

但必须明确：

- 它只用于说明性展示或兼容旧结构
- 不能直接作为真实设备执行输入

### 2. 唯一可面向设备执行层演进的结构是智能面罩控制协议

当前执行前控制协议是：

- `mask_control.v1`

它必须至少包含：

- `treatment_plan_id`
- `user_public_id`
- `overall_severity`
- `global_settings`
- `zone_led_settings`
- `execution_channel`

### 3. 设备接入必须通过独立控制层

未来真实硬件接入时：

- 前端不能直接调用硬件 SDK
- 路由层不能直接写硬件逻辑
- 只能由 `infrastructure/hardware/` 扩展真实控制器
- 设备执行结果必须回写治疗记录层

## 十三、架构变更要求

以下变化一旦发生，必须同步更新本文件：

- 页面数量变化
- 核心数据流变化
- 治疗方案生成逻辑变化
- 控制协议字段变化
- 新增或删除共享资料库目录
- 硬件抽象发生变化

本文件对项目内所有后续代码修改具有约束力。

# 架构规范文档 - VLM皮肤识别与光疗系统

## 一、总体目标

本项目用于构建一个生产级的「VLM 皮肤状态识别与光疗方案系统」，支持：

- 多用户档案管理与自动分配用户 ID
- 基于云端视觉模型的面部皮肤状态识别
- 每日图像与评估结果归档，形成用户皮肤状态库
- 基于专业知识库与案例样本库生成治疗建议
- 前后端彻底分离
- 硬件控制能力预留，后续接入温度、光照、加湿等实体设备接口

## 二、分层与目录

所有代码必须按以下结构组织：

### `backend/`

后端服务与 CLI，仅负责业务逻辑、模型调用、数据持久化、文件归档与硬件接口抽象。

#### `backend/app/api/`

HTTP API 层。

- 只负责请求校验、调用服务、返回响应
- 不得直接包含复杂业务逻辑

#### `backend/app/core/`

系统核心配置。

- 环境变量读取
- 路径配置
- 启动初始化

#### `backend/app/domain/`

领域层。

- 核心业务服务
- 领域规则
- 模型输出归一化逻辑

#### `backend/app/infrastructure/`

基础设施层。

- 数据库访问
- 文件存储
- 外部模型供应商适配
- 硬件控制接口

#### `backend/app/schemas/`

输入输出 Schema。

- API 请求/响应模型
- CLI 输出结构

#### `backend/tests/`

后端测试代码。

### `frontend/`

前端应用，仅负责界面、交互、状态管理与 API 调用。

#### `frontend/src/app/`

应用壳、路由与启动逻辑。

#### `frontend/src/pages/`

页面级组件。

#### `frontend/src/features/`

按业务功能拆分的前端模块。

#### `frontend/src/shared/`

共享 UI、API 客户端、样式与通用工具。

### `common/`

前后端共享的、与具体运行时无关的资产。

- 合同文件
- 知识库
- 提示词模板
- 治疗案例样本

### 根目录脚本

根目录允许放置开发与启动脚本，例如：

- `start-dev.ps1`
- `start-dev.bat`

这些脚本只负责开发环境启动、依赖检查与进程编排，不承载业务逻辑。

## 三、职责分离原则

### 统筹修改原则

- 所有功能修改必须先从前端、后端、共享合同三个维度做统筹规划，再开始实现
- 先确认数据流、接口契约、状态影响范围，再进行代码修改
- 禁止只改单侧实现而让另一侧被动适配
- 涉及 UI 交互变更时，必须同步检查 API、领域服务、持久化与文档是否需要调整

### 前端与后端

- 前端不得直接访问数据库、文件系统或模型 SDK
- 后端不得依赖前端页面或组件实现
- 后端能力必须可以通过终端 CLI 独立使用

### API 层与业务层

- Router 只负责协议转换
- 业务判断必须放在 `domain/services`
- 外部供应商调用必须放在 `infrastructure/llm`

### 共享资产

- 可被前后端共同理解的知识与合同，放在 `common/`
- 语言特定代码不得混放到 `common/`

## 四、数据流规范

### 面部测评流程

1. 用户在前端选择或创建用户
2. 前端上传图像或通过摄像头拍照，并选择模型供应商与界面语言
3. 后端保存原始图像到归档目录
4. 后端组合提示词、知识库与案例样本
5. 后端调用视觉模型完成结构化评估
6. 后端将评估结果与图像路径写入状态库
7. 前端展示整体状态、分区状态与治疗建议

### 面部档案流程

1. 前端请求某个用户的历史评估列表
2. 后端从数据库读取评估摘要与详情
3. 前端按时间线展示每日状态与对应图像

## 五、命名与约束

### 用户标识

- 数据库主键可使用内部自增 ID
- 对外展示必须使用自动分配的业务 ID
- 业务 ID 格式统一为：`USR-0001`

### 面部分区命名

必须使用以下标准名称：

- `Forehead Zone`
- `Periorbital Zone`
- `Nasal Zone`
- `Left Malar Zone`
- `Right Malar Zone`
- `Perioral Zone`
- `Mandibular/Chin Zone`
- `Jawline Zone`

### 问题程度

统一为：

- `low`
- `medium`
- `high`

### 模型供应商标识

统一为：

- `gemini`
- `qwen`

### 语言标识

统一为：

- `en`
- `zh`

### 多语言规则

- 内部结构化字段统一使用英文标准码
- 前端根据当前语言对标准码做显示映射
- 可读文本必须支持中英文输出
- 与硬件控制直接相关的字段不得使用自由文本作为控制参数

## 六、文档与扩展

新增功能时，必须说明：

- 该功能位于哪一层
- 为什么放在该层
- 新增了哪些依赖
- 是否影响既有数据流

架构或目录变更时，必须同步更新本文件。

## 七、基础设施

### 环境变量

敏感信息只允许存放在 `.env` 中，严禁硬编码进源码。

### 持久化

- 结构化业务数据：SQLite
- 原始图像文件：文件系统归档
- 后续可平滑替换为 PostgreSQL + 对象存储

### 硬件控制扩展

必须保留以下能力抽象：

- 光色控制
- 温度控制
- 加湿控制
- 治疗时长控制
- 治疗任务下发

硬件相关数据必须使用结构化命令协议输出，至少包含：

- `zone_code`
- `light_type_code`
- `temperature_celsius`
- `duration_minutes`
- `humidification_enabled`
- `safety_profile`

当前阶段允许使用 Mock 实现，但接口必须稳定。

- 摄像头拍照入口

- 面部档案浏览
